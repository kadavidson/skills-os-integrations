Built with the Adaptive AI team, May 2026. The design team's implementation guide for the AI-first PDLC RFC.

01 — What we tried

The sprint — and what broke

We ran a 2-month sprint with no traditional handoffs. Design and engineering worked in daily cycles, decisions were made in standups, and a live prototype was the primary artifact. It moved fast. It also created real problems: rework, no source of truth, design craft under pressure, and leadership feedback arriving after work was already done.

This site proposes a better version — one that keeps the speed and kills the friction.

02 — The proposed process

Two tracks. One goal. No waiting.

The key insight: Eng doesn't wait for design to finish. Design doesn't wait for eng to start. They run in parallel — and merge when we get into core UX flows.

⚙️ENG TRACK — starts immediately
Eng 01
Core Capabilities
  • Builds infrastructure, APIs, data models from day one
  • Any area design hasn't flagged = eng can start immediately
  • No waiting for design to "finish"
Eng 02
Feasibility Spikes
  • When design flags an Eng question, eng runs a timebox spike (2–4 hrs max)
  • Output: yes / yes-with-cost / no — not a full build
  • Feeds directly back into design's options
🎨DESIGN TRACK — runs in parallel
Design 01
AI Prototype + Flag

Build a working prototype with Claude Code or Cursor — fast. Goal: feel the experience, surface real questions. As you build, tag every open question:

🟠 UX — best experience? 🔵 Eng — feasible / at what cost? 🟣 Biz — what constraint applies?

A flag = needs divergence before dev builds it. No flag = dev can start. The prototype is a question generator, not the source of truth.

Design 02
Diverge by Track

Each flag becomes a parallel mini-exploration with the right owner. Protected time — not resolved in standup. For UX flags, the goal is genuine divergence: multiple variants, not a single answer dressed up as options.

TrackOwnerOutput
🟠 UX Designer Multiple variants with documented tradeoffs — FigJam for discussion, screenshots of key states, links to prototypes
🔵 Eng Engineer spike yes / yes-with-cost / no
🟣 Biz PM / biz owner What are we optimizing for?
🔗 Align & Lock — where the tracks merge
All three outputs come together
  • Options shared async with leadership before the sync
  • One named decision-maker per open question
  • Decision documented in the repo before UX flows are built
  • After this: locked. A change = new flag, new cycle.
The contract

Things can change. That's expected. But the source of truth updates before dev acts on it. A decision made in Slack doesn't exist until it's in the repo.

Decisions land in the initiative's prds/ folder — the same behavior contract the RFC's drift-detection tooling runs against. Figma frames are linked from the PRD for specific visual call-outs. The repo is the source of truth; Figma is a linked reference.

Soon after a decision is locked: prototypes and Figma frames for the affected flow are updated to match. No artifact should show a superseded state.

03 — Post-merge

Design in code

After Align & Lock, designers stay in the work through shipping — using AI tools to make changes directly in the codebase.

What's changing about Figma's role

Exhaustive hi-fi Figma is no longer the handoff. Figma is a collaboration space — a place to explore options during the Diverge phase, and to capture specific visual call-outs that the prototype or design system can't express on their own (a layout treatment, an edge state, a motion intent). Those frames get linked directly from the PRD in the repo. Figma doesn't own the behavior contract — the repo does.

Figma is for
  • Exploring UX options (Diverge phase)
  • Screenshots of specific visual decisions linked from the PRD
  • Design system component work
Figma is not for
  • Exhaustive hi-fi handoff
  • The primary artifact for dev to build from
  • The source of truth

The split — what designers own in code

Designer owns in code Engineer owns Shared (AI-assisted)
All copy / UX writingComponent architectureVisual component tweaks already in PRD
Color tokens, spacing, typeData + state managementAnimation timing
Image / icon swapsAPI integrationResponsive breakpoints
Minor layout adjustmentsComplex interactions—

The workflow

1

Decision is locked in Align & Lock — captured in prds/ in the repo

2

Designer checks: is this in my zone?

3

If yes: use Cursor or Claude Code to make the change

AI writes the code, designer drives

4

Submit PR — same process as dev

5

Eng reviews for technical correctness, designer reviews for visual accuracy

6

Merge. PR links back to the relevant PRD file and any Figma frame for that specific call-out.

Why this matters

Designers staying in the work through shipping reduces round-trips, eliminates the prototype/product duplication problem, and means design craft shows up in the final product — not just in Figma.

04 — Running commitment

The daily log

This isn't a phase — it's a commitment that runs through the entire process. After every key moment:

🔒A decision is locked
✅A flag is resolved
📄A PRD changes
🚀A PR merges
  1. The repo is updated — the relevant prds/ file reflects the current state of decisions
  2. Prototypes and Figma frames for the affected flow are updated to match — no artifact should show a superseded state
  3. Figma frames with visual call-outs are linked from the PRD
  4. AI summarizes the day's changes from Slack + PRs into a digest — this is the RFC's daily digest agent. Design plugs into the same loop.
  5. Someone approves the digest (5 min) — it becomes the durable record

What the source of truth contains (all in the repo)

📋
prds/ files
Behavior contracts — what was decided, acceptance criteria, edge cases
📐
Figma links
Specific visual call-outs linked from PRD — not exhaustive hi-fi
📝
Decision log
What changed, who decided it, why — auto-generated from PRs + Slack
🚩
Flag tracker
Open questions + their current status
The rule

Leadership shouldn't need to attend every sync to stay current. They should be able to open the repo. If they can't — the log isn't being kept.

05 — Team reflections

What we'd keep. What we'd change.

✓ What worked

Seeing the product in a real working prototype early — you can feel UX problems that Figma can't show

Deep cross-functional collaboration — design, eng, and PM in the same room daily

Speed of iteration when the team was aligned

Prototype-first allowed testers to provide feedback quickly

— Ina

Great to see the product being built out fast

— Bianca
↻ What we'd change

No source of truth. Decisions were made in standups and Slack — often never written down. Led to rework and confusion for anyone not in the room.

— Esteban, Bianca, Ina

Design craft under pressure. With decisions made on the fly, no time for exploration, research, or peer review. Quality suffered.

— Esteban

Leadership feedback arrived late. By the time leadership weighed in, work was already done. Painful to change.

— June, Esteban

Onboarding was hard. No doc to read meant new team members had no way to get context async.

— June, Bianca

Prototype and product duplicated work. Design was updating the prototype while eng was building the same thing.

— June

Siloed from the rest of the org. Limited alignment with platform design and broader teams.

— Ina, Bianca
06 — Still unresolved

What's still open

This is a proposal, not a finished playbook. These questions need the whole team to answer.

01 How do we handle leadership feedback that arrives after a decision is locked? Is there a defined process for deferring vs. acting?
02 What does the PR review process look like for design code contributions — who owns approval?
03 How do we stay connected to the broader design org and design reviews when moving this fast?
04 What's the minimum viable daily log — what must be in the repo vs. what can stay in Slack?
05 How do we define "MVP scope" vs. future iteration clearly enough that the team isn't second-guessing constraints?
06 When does a design decision warrant a Figma frame in the PRD vs. a prose description? Who decides?

This is a proposal, not a playbook. Add your questions, flag what's missing, and let's build the next version together.
